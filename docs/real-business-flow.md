## 零、朗姿智能工牌系统核心定位（最重要）

**这个系统不是即时作业工具，而是事后复盘工具。**

具体来说：
- 咨询师和医生的**核心工作（面对面咨询、录音）发生在线下**，由工牌硬件和手机端承载。
- 这个系统的价值在于：录音完成并经过 AI 处理后，让咨询师/医生能够**事后回看和复盘**。
- 复盘的内容包括：
  - 近期接待了哪些客户
  - 每次接待关联的录音是哪段
  - 录音中提取的关键信息（客户需求、标签等）
  - 话术质检结果（是否正确沟通引导、有无违规）

这意味着整个系统的**两个时间阶段**是分离的：

```
【第一阶段：线下作业（当天发生）】
  前台建单 → 分配 → 录音 → 咨询/面诊 → 可能成交 → 补单
  ↓ 数据异步流入
【第二阶段：事后复盘（一段时间后）】
  咨询师/医生在企微工作台或 PC 后台中：
  查看接待记录 → 查看关联录音 → 查看 AI 提取结果 → 查看质检结果
```

## 二、系统的四个端

| 端 | 用户 | 核心目的 | 使用时机 |
|----|------|---------|---------|
| **微信小程序** | 咨询师、医生 | 控制工牌录音（开始/结束），纯遥控器 | 线下服务当天 |
| **企微工作台** | 咨询师、医生 | 列表视角复盘：我的客户列表、接待记录列表、录音列表、移动端关联操作 | 事后 |
| **PC 后台** | 前台、管理员 | 创建到诊单、分配咨询师；标签/热词/模板/质检配置 | 当天 + 日常管理 |

### 关键认知：当前企微入口的正确定位

当前企微入口的定位是**事后复盘工具**，不是现场作业工具。

现阶段主入口是**企微工作台 H5**：
- 咨询师/医生在企微中打开工牌应用
- 查看自己的接待记录、录音、AI 分析结果、质检结论
- 进行录音与到诊单的人工关联
- 必要时跳转到 PC 后台做更重的管理操作

企微入口**不需要**：创建到诊单、采集需求标签、发起接待。
企微入口**需要**：展示历史记录、AI 分析结果、质检结果、关键录音片段回放与关联操作。

| 角色 | 使用端 | 核心动作 |
|------|--------|---------|
| 前台工作人员 | PC 后台 | 创建到诊单、分配咨询师、查看到诊状态 |
| 咨询师 | 小程序 + 企微工作台 | 小程序：控制录音；企微：事后复盘 + 补齐到诊单信息 |
| 医生 | 小程序 + 企微工作台 | 小程序：控制录音；企微：事后复盘 + 补齐到诊单信息（含充值字段） |
| 管理员/运营 | PC 后台 | 机构管理、设备管理、人员/权限管理、热词/词包管理、AI 模型管理、质检维度配置、标签配置、风控规则、统计看板、报告导出 |

## 四、核心实体

| 实体 | 说明 | 关键关系 |
|------|------|---------|
| **客户 (customer)** | 到院客户，可同时是企微外部联系人 | 身份标识：手机号为主，external_userid 为辅 |
| **到诊单 (visit)** | 一次到院服务的主单据，由前台创建 | 一个客户可有多张到诊单；到诊单通过对话片段(segment)关联录音；包含"充值本金"和"充值增值金"字段，有值即表示成交；可能无关联录音（忘记录音） |
| **录音 (recording)** | 工牌设备产生的原始录音文件 | 属于某个员工和设备；一段录音可能包含与多个客户的对话；可能缺失（忘记录音）；上传时携带设备 ID + 员工 ID 标识来源 |
| **转写文本 (transcript)** | ASR 转写产生的带时间戳文本 | 一段 recording 对应一份 transcript；包含说话人分离、时间戳；是片段拆分和 AI 分析的输入 |
| **对话片段 (segment)** | 一段录音中识别出的单次客户对话 | 属于某段录音，有起止时间；与到诊单的关联发生在片段级别，不是录音级别 |
| **AI 总结 (ai_summary)** | 对对话片段转写文本的 AI 分析结果 | 一个片段对应一份总结 |
| **质检结果 (quality_check)** | 对对话片段中话术的质检评分 | 一个片段对应一组质检结果（每个维度一条得分记录） |
| **标签结果 (tag_result)** | AI 提取或人工标注的标签 | 可挂在客户/录音/到诊单下 |
| **员工 (staff)** | 系统用户 | 有角色（前台/咨询师/医生/管理员），可绑定工牌设备，属于某个机构 |
| **机构 (organization)** | 集团下的独立运营机构 | 员工归属机构；数据按机构隔离；支持集团-机构两级结构 |
| **工牌设备 (device)** | 录音硬件设备 | 设备编号唯一；绑定到某个员工；有状态（出厂/离线/在线）、充电状态、电量、GPS 精度等属性 |
| **AI 模型 (ai_model)** | 一组质检维度和检查点的集合，代表一套完整的质检标准 | 一个模型包含多个质检维度，每个维度包含多个检查点；可启用/停用；评分时使用当前启用模型下的维度和检查点 |
| **热词词包 (hotword_pack)** | 词汇集合，用于 ASR 增强和质检辅助 | 分类：优质话术词库（个人/公共）、竞品机构词包、专属词包；可配置推送和强转 |
| **风控规则 (risk_rule)** | 定义违规/风险行为的触发条件 | 关联质检维度中的违规类检查点；触发后生成风控记录 |
| **风控记录 (risk_record)** | 员工触发风控的记录 | 关联员工、到诊单、对话片段；记录触及文本和风险点 |
| **操作日志 (audit_log)** | 系统操作审计记录 | 记录用户的关键操作（登录、数据变更等）；用于安全审计和行为分析 |

## 五、数据流定义

### 5.1 到诊单的生命周期

```
创建(前台)
  → 已分配咨询师(前台分配)
    → 咨询中(咨询师开始录音)
      → 咨询完成(咨询师结束录音)
        ├→ 需面诊(分配医生)
        │    → 面诊中(医生开始录音)
        │      → 面诊完成(医生结束录音)
        │        ├→ 已成交(到诊单中填写充值本金+充值增值金)
        │        └→ 未成交(充值字段为空)
        └→ 未成交
    → 待补录(事后补信息)
      → 已完成(信息补齐、录音关联完毕)
```

### 5.2 录音的数据流

```
产生：咨询师/医生在手机端点"开始录音" → 工牌设备录音 → 点"结束录音"
  ※ 异常情况：忘记开始（无录音）、忘记结束（超长录音跨多个客户）、
    中途断开（多段短录音）、开始太晚/结束太早（不完整）

上传：工牌 → (网络) → 服务器(MinIO)，网络不畅时本地缓存、恢复后续传
  ※ 上传请求须携带：设备 ID + 员工 ID，用于标识录音来源

转写：录音上传完成后，系统主动调用 ASR API 提交转写任务 → 轮询/回调获取结果 → 带时间戳的文字结果入库

异步任务状态通知：
  → 前端通过轮询 API 获取任务状态（转写中/转写完成/分析中/分析完成）
  → 录音和片段记录上维护 processing_status 字段
  → 列表页和详情页按状态筛选和展示进度

片段拆分：AI 分析转写文本 → 识别对话边界 → 拆分为若干对话片段(segment)
  ※ 正常情况：一段录音 = 一个片段
  ※ 忘记结束：一段录音 → 多个片段
  ※ 每个片段内部可能有多个说话人（客户、亲友、其他工作人员）

关联：每个片段与到诊单关联（系统推荐 + 人工确认）

分析：对每个已关联的片段执行：
  → AI 角色识别：区分咨询师/医生的话 vs 客户的话 vs 第三方的话
  → AI 总结：基于客户的话提取需求、标签
  → 质检：基于咨询师/医生的话评估话术质量
  → 结果入库

消费：咨询师/医生在企微工作台或 PC 后台查看片段级的分析结果（复盘）
```

### 5.3 复盘数据的消费流

```
企微工作台（面状的复盘）：
  我的接待记录列表 → 按时间/客户/成交状态筛选
  我的录音列表 → 按状态(待处理/已分析)筛选
  我的质检结果 → 查看质检评分趋势

PC 后台（管理与深度复盘）：
  查看录音、客户、到诊、分析结果与配置项
  支持更完整的筛选、导出和管理操作
```

### 5.4 质检数据流

```
配置阶段（管理员，PC后台）：
  创建 AI 模型（如"医美咨询标准 V1"），一个模型 = 一套质检标准
  → 在模型下定义质检维度（如"需求挖掘"、"产品介绍"、"异议处理"、"违规话术"等）
  → 每个维度下定义若干检查点（如"是否询问客户预算"、"是否介绍术后护理"等）
  → 每个检查点有名称、描述/判定指引
  → 模型、维度和检查点可启用/停用、可排序
  → 评分时使用当前处于"启用"状态的模型

评分阶段（AI 自动，录音转写完成后触发）：
  前置：AI 先对转写文本做角色识别（区分咨询师/医生 vs 客户 vs 第三方）
  输入：咨询师/医生的话术部分 + 对话上下文 + 当前启用的维度及检查点列表
  处理：AI（LLM）对每个检查点逐项判断 → 命中 / 未命中
  输出：每个检查点的命中状态 + 依据原文片段
  汇总：维度得分 = 该维度下命中检查点数 / 总检查点数（或加权汇总）
  → 写入 quality_check_result（维度级）+ quality_check_detail（检查点级）

消费阶段（咨询师/医生，企微工作台）：
  查看某段录音的各维度得分
  展开维度查看每个检查点的命中/未命中状态及引用原文
  AI 质检打分体现录音上下文关联（二期：结合前后文语境评分，而非孤立句子）
  （管理员可在后台查看全员质检概览）

风控触发（与质检评分同步执行）：
  → 质检中"违规话术"等风控相关维度的检查点如果命中
  → 自动生成风控记录 (risk_record)，写入触发员工、关联到诊单、触及文本
  → 管理员可在后台查看机构整体/员工个人的风控触发次数和明细

报告阶段（管理员，PC后台）：
  → 按机构/员工维度生成质检报告表
  → 支持导出（Excel/PDF）
```

### 5.5 设备管理数据流

```
PC 后台 → 设备管理页面：

设备列表：
  展示所有工牌设备：设备编号、使用人、状态（出厂/离线/在线）、
  充电状态、电量、GPS 精度
  → 支持按机构名称、设备编号、设备状态、GPS 模式筛选

设备操作：
  → 绑定：将设备分配给某个员工（一个设备同一时间只绑定一个人）
  → 解绑：解除设备与员工的绑定关系
  → 编辑：修改设备信息（备注、GPS 模式等）

设备状态更新：
  → 设备通过网络定期上报心跳（在线/离线/电量/充电状态/GPS）
  → 系统接收心跳数据更新设备状态
  → 定时任务（APScheduler 或 cron 触发 dramatiq 任务）检查心跳超时，超时标记为离线
```

### 5.6 AI 模型管理数据流

```
PC 后台 → AI 模型管理页面：

模型定义：
  → 一个 AI 模型 = 一套完整的质检标准（维度 + 检查点的集合）
  → 例如："医美咨询标准 V1"包含"需求挖掘"、"产品介绍"、"违规话术"等维度
  → 每个模型包含：名称、描述、状态（启用/停用）、更新时间
  → 质检维度和检查点挂载在模型下，而非独立管理

模型操作：
  → 新增模型：手动创建或通过规则导入
  → 编辑模型：修改规则、调整检查点
  → 搜索模型：按名称/状态筛选
  → 启用/停用：控制模型是否参与当前质检评分
  → 规则导入：支持从文件批量导入质检规则

模型与质检的关系：
  → 质检评分时，使用当前处于"启用"状态的模型对应的维度和检查点
  → 不同机构可配置不同的启用模型（二期）
```

### 5.7 热词与词包管理数据流

```
PC 后台 → 热词管理页面：

词库分类：
  1. 优质话术词库
     → 个人自定义词库：员工自行添加的话术关键词
     → 公共话术词库：团队共享的优质话术关键词
  2. 竞品机构词包：竞品品牌名称集合（用于识别客户提及竞品场景）
  3. 专属词包：业务专属词汇（医美项目名、设备名等，提升 ASR 识别准确率）

词包操作：
  → 新增词库/词包
  → 编辑词包内容（增删词汇）
  → 搜索词包
  → 启用/停用

推送配置：
  → 配置哪些词包推送给 ASR 服务以提升转写准确率
  → 配置推送范围（全机构 / 指定团队 / 指定员工）

强转配置：
  → 定义需要强制转写的关键词（即使 ASR 置信度低也保留）
  → 典型场景：机构名、项目名等必须准确识别的词汇
```

### 5.8 机构与人员管理数据流

```
PC 后台 → 机构/人员管理页面：

机构管理：
  → 集团下可包含多个机构
  → 机构信息：名称、编码、状态（启用/停用）
  → 数据隔离：员工只能看到本机构的数据，集团管理员可看全部

人员管理：
  → 列表展示：账号、姓名、手机号、角色、所属机构、职位、状态
```

## 七、待确认的关键问题

### Q1: 手机端的载体是什么？
✅ **已确认**：
- 载体：**微信小程序**，免安装
- 功能定位极窄：**只控制工牌录音**（开始/结束）
- 小程序上看不到到诊单列表（分配通过电话通知，不走系统）
- 录音按键需**隐蔽且易操作**，尽量不让客户察觉
- 咨询师浏览复盘信息走企微工作台，不走小程序

### Q2: 工牌设备如何与手机端通信？
⚠️ **初步方向**：
- 蓝牙优先，WiFi 备选
- 具体协议和交互细节待硬件方确认
- 对我们系统的影响：小程序需要调用蓝牙/WiFi API 与工牌通信，但这部分逻辑封装在小程序内部，不影响后端和企微工作台/PC 后台的设计

### Q3: 录音如何与到诊单关联？
✅ **已确认**：
- 方案：**事后自动推荐 + 人工确认**

**关联难点（已知约束）：**
1. 到诊单分配通过电话通知，小程序上看不到到诊单列表 → 录音时无法主动绑定
2. 咨询师可能接待非分配给自己的客户（互相帮忙）→ "录音人 = 到诊单分配人"不成立
3. 实际录音时间 ≠ 到诊单登记时间 → 时间匹配不精确
4. 客户在录音中说的姓名可能不真实 → 录音内容提取姓名匹配不可靠
5. 忘记点"开始录音" → 与某个客户的对话完全没有录音，到诊单无法关联到任何录音
6. 忘记点"结束录音" → 一段录音包含与多个客户的对话，录音和到诊单不是 1:1 而是 1:N
7. 中途意外断开 → 一次接待被切成多段短录音，到诊单和录音是 1:N
8. 开始太晚/结束太早 → 录音不完整，只覆盖部分对话
9. 对话非 1V1 → 客户可能有亲友陪同，其他工作人员可能中途插入对话，一段对话中存在多个说话人

**结论：录音和到诊单之间是多对多关系，且粒度可能不是"整段录音"而是"录音片段"。**

**关联策略（分三步）：**

**第一步：录音转写后，AI 自动识别对话边界**
- 长录音的转写文本中，AI 尝试识别"不同客户对话"的分界点
  - 注意：一次客户对话中本身就可能有多人说话（亲友陪同、其他工作人员插话），说话人数变化≠客户切换
  - 可靠的切分信号：长静默段（几分钟沉默）、明确的场景转换语句（如"下一位"、"您好请坐"）、话题完全不相关的跳变
  - 不可靠的信号：说话人数量变化、称呼变化（同一客户的亲友也会被称呼）
- 将一段长录音标注为若干**对话片段 (segment)**，每个片段有起止时间
- 如果录音只包含一个客户的对话，则只有一个片段
- 对于边界不确定的情况，AI 标注为"疑似分界"，留给人工确认

**第二步：系统对每个片段自动推荐候选到诊单**
- 可用信号：同一天 + 同一员工 + 时间窗口 + 转写内容关键词
- 推荐结果展示在工作台"待关联"列表中

**第三步：人工确认绑定**
- 咨询师/医生（或管理员）在工作台确认：这个片段对应哪张到诊单
- 也可标记为"无法识别"或"无对应到诊单"
- 对于忘记录音导致无录音的到诊单，标记为"缺失录音"

### Q4: 到诊单在 PC 后台由前台创建，还是从第三方 HIS 系统同步？
✅ **已确认**：
- 前台在 PC 后台创建到诊单（填写客户基础信息、到院时间等）
- 后续由咨询师和医生共同补齐字段内容（到诊目的、咨询情况、成交信息等）
- 不对接第三方 HIS 系统，到诊单完全在本系统内闭环
- 补齐操作发生在企微工作台中（事后）

### Q5: 充值单的详细字段？
✅ **已确认**：
- 不需要独立的充值单实体
- 成交信息直接记录在到诊单的两个字段中："充值本金"和"充值增值金"
- 判断是否成交的逻辑：这两个字段有值即为成交，为空即为未成交

### Q6: ASR 转写服务的对接方式？
✅ **已确认**：
- 拉模式：录音上传完成后，我们主动调用 ASR API 提交转写任务
- 轮询或回调获取转写结果（取决于 ASR 服务商的具体接口，但主动权在我们这边）
- 异步任务流：录音上传完成 → dramatiq 任务调用 ASR API → 获取结果写入数据库 → 触发后续 AI 分析

### Q7: 质检规则由谁定义？粒度是什么？
✅ **已确认**：
- 管理员在 PC 后台定义质检维度（如"需求挖掘"、"产品介绍"、"异议处理"、"违规话术"等）
- 每个维度下定义若干**检查点**（如"需求挖掘"下有"是否询问客户预算"、"是否了解客户既往史"等）
- 录音转写完成后，AI 对每个检查点逐项判断**命中/未命中**
- 维度得分由该维度下检查点的命中情况汇总得出
- 咨询师/医生在企微工作台查看每段录音的各维度得分及检查点明细进行复盘

### Q8: 企微移动端入口中除了查看，咨询师/医生是否需要做任何写操作？
✅ **已确认**：
- 当前正式移动入口是**企微工作台**
- 企微工作台支持**轻量写操作**，复杂编辑走 PC 后台
- 企微工作台可做的写操作（轻量）：
  - 补齐到诊单的简单字段
  - 确认录音片段与到诊单的关联
  - 给客户补标签、写简短备注
- 企微工作台不做的写操作（复杂，走 PC 后台）：
  - 创建到诊单（前台 PC 后台）
  - 大量字段的表单编辑
  - 质检维度/标签包等后台配置
